# Agent Journals Design

An automated journaling system for Claude Code sessions, producing structured records in Obsidian that support session-level capture, longitudinal thread tracking, and periodic reflection.

## Architecture

Three folders in `~/obsidian/Agent Journals/`, three components producing content.

```
Agent Journals/
  Sessions/       ← one per Claude Code session, auto-generated by /journal skill
  Threads/        ← one per line of work, updated by /reflect skill
  Weekly/         ← one per week, generated by scheduled agent
```

### Components

1. **`/journal` skill** — writes a session journal to `Sessions/`. Triggered by a session-end hook. Reads the session transcript from `~/.claude`, writes via `notesmd` CLI.

2. **`/reflect` skill** — reads session journals and existing thread files, creates or updates thread documents in `Threads/`. Available on-demand. Can also be triggered after `/journal` completes.

3. **Weekly digest (scheduled)** — a cron-triggered agent that runs weekly (e.g., Sunday evening). Reads all sessions and threads from the week, writes a digest to `Weekly/`.

### Data Flow

```
Session ends → hook fires → claude /journal → Sessions/*.md
User types /reflect → reads Sessions/ → updates Threads/*.md
Cron (Sunday) → reads Sessions/ + Threads/ → writes Weekly/*.md
```

### Tool: notesmd CLI

All writes go through `notesmd` (installed at `~/bin/notesmd`, default vault "Nat"). The Obsidian vault is at `~/Library/Mobile Documents/iCloud~md~obsidian/Documents/Nat` (symlinked to `~/obsidian`).

Key commands:
- `notesmd create "Agent Journals/Sessions/note-name" --content "text"` — create
- `notesmd create "Agent Journals/Threads/note-name" --content "text" --append` — append
- `notesmd print "Agent Journals/Sessions/note-name"` — read
- `notesmd search-content "term"` — search

## Session Journal Format

**Filename:** `YYYY-MM-DD - Title.md` with letter suffix for multiple sessions per day (`a`, `b`, `c`).

**Every session gets a journal**, regardless of length or significance. Trivial sessions get short entries. The reflection layer filters signal from noise.

### Frontmatter

```yaml
---
session: 81f8ee08
date: 2026-03-22
project: /Users/gnat/Source
threads:
  - Meeting Transcription Pipeline
  - Ollama Setup
tools_installed:
  - OpenOats
  - Ollama (qwen3:8b)
  - FluidAudio CLI
tags:
  - transcription
  - local-llm
  - audio
---
```

Fields:
- `session` — Claude Code session ID (from `~/.claude`)
- `date` — session date
- `project` — working directory path
- `threads` — agent-proposed thread names (editable by user; next `/reflect` pass regroups accordingly)
- `tools_installed` — software installed or configured during the session
- `tags` — topic tags for Obsidian Dataview queries and filtering

### Body Sections

```markdown
# YYYY-MM-DD — Title

## Summary
One to three sentences. What happened and why it matters.

## What Happened
Narrative of the session — key decisions, approaches tried, pivots made.
Links to relevant files and PRs.

## What Worked
- Bullet points of things that went well

## What Didn't
- Bullet points of friction, failures, dead ends

## Learnings
- Durable takeaways from this session

## Improvement Ideas
- Concrete suggestions for new skills, commands, workflow changes
```

## Thread Document Format

**Filename:** Thread name as title, e.g., `Meeting Transcription Pipeline.md`.

Threads are lines of work that span multiple sessions. They are intent-based, not path-based — a thread groups sessions by what you were trying to accomplish, not which directory you were in. Thread names are proposed by the `/journal` skill in session frontmatter. Users can edit thread names; the next `/reflect` pass regroups.

### Structure

```markdown
---
thread: Meeting Transcription Pipeline
status: active
started: 2026-03-22
last_updated: 2026-03-24
sessions:
  - "[[2026-03-22b - OpenOats Ollama Meeting Transcription]]"
  - "[[2026-03-24a - Transcript Cleanup Improvements]]"
---

# Meeting Transcription Pipeline

## Current Status
Working pipeline: extract audio → transcribe (FluidAudio) → diarize → merge →
LLM cleanup (4 of 5 passes working). Whole-document boundary detection pass
still timing out on long transcripts.

## Open Questions
- How to handle transcripts >19K tokens in a single LLM pass?
- Should the pipeline be packaged as a skill or a standalone script?

## Key Decisions
- FluidAudio over whisper.cpp (better diarization integration)
- qwen3:8b for cleanup (with /no_think)

---

## 2026-03-24 Update
Improved cleanup pass ordering. Tested on a second recording — speaker
identification accuracy improved with --attendees flag. Still need to solve
the long-document pass.

## 2026-03-22 Update
Initial pipeline built. Transcribed 96-minute Zoom recording. Discovered qwen3
thinking mode issue. 4 of 5 cleanup passes working. Scripts in ~/Downloads/ —
should relocate.
```

### Design Decisions

- **Newest at top.** Current Status and Open Questions are always the first thing you see. Dated updates accumulate below as a changelog.
- **Obsidian wikilinks.** Sessions listed as `[[wikilinks]]` in frontmatter so Obsidian's graph view and backlinks connect threads to sessions.
- **Status field.** One of: `active`, `paused`, `complete`, `abandoned`. The reflection agent infers this from session recency and content. User can override.
- **`/reflect` rewrites top sections, appends to history.** Each reflection pass updates Current Status, Open Questions, and Key Decisions based on new session data, then prepends a new dated update to the changelog section.

## Weekly Digest Format

**Filename:** `YYYY-WNN.md` (ISO week number).

Generated by a scheduled agent running weekly (e.g., Sunday evening). Reads all sessions from the week plus current thread state.

### Structure

```markdown
---
week: 2026-W12
dates: 2026-03-16 to 2026-03-22
session_count: 6
active_threads:
  - "[[Meeting Transcription Pipeline]]"
  - "[[Obsidian Agent Infrastructure]]"
  - "[[Ontempo Warehouse Bugs]]"
---

# Week 12: Mar 16–22

## Overview
Heavy infrastructure week. Set up the Obsidian vault and agent tooling
(notesmd, Agency plugin), then pivoted to a meaty transcription pipeline
build. One Ontempo bug investigation. 6 sessions across 4 projects.

## Thread Progress
- **Obsidian Agent Infrastructure** — vault structure created, notesmd working,
  journals flowing. (3 sessions)
- **Meeting Transcription Pipeline** — end-to-end pipeline working with caveats
  (long doc pass, thinking mode). (1 long session)
- **Ontempo Warehouse Bugs** — RFID barcode issue identified, audit recommended.
  (1 session)
- **Skill Safety Review** — started, unresolved. (1 session)

## Recurring Patterns
- Local LLM configuration keeps requiring debugging (model pulling, timeouts,
  thinking modes)
- Tools installed in ~/Downloads/ rather than proper locations — cleanup needed

## Improvement Proposals
- **Create /transcribe skill** — the pipeline is reusable but currently
  scattered scripts (surfaced 1x)
- **Ollama pre-flight check** — verify model is pulled and responsive before
  long operations (surfaced 1x)
- **Skill review sandbox** — the meta-problem of reviewing skills safely
  remains open (surfaced 1x)

## Sessions This Week
- [[2026-03-17 - Superpowers Plugin Installation]]
- [[2026-03-20 - Obsidian-to-PDF Typst Backend]]
- [[2026-03-21 - Obsidian Vault Setup & CLI Tools]]
- [[2026-03-22a - Skill Safety Review]]
- [[2026-03-22b - OpenOats Ollama Meeting Transcription]]
- [[2026-03-23 - Ontempo Warehouse RFID Barcode Bug]]
```

### Design Decisions

- **Recurring Patterns** is the highest-value section. Surfaces friction you wouldn't notice in individual sessions.
- **Improvement Proposals track frequency.** An idea surfacing 3x across weeks is screaming to be acted on.
- **Thread Progress is narrative, not a list.** One-line summary per thread of what moved and what's stuck.
- **Overview tone matches the user's voice** (see Voice section below).

## Voice

All generated text must match the user's natural writing voice. The skill prompts will include voice calibration instructions.

### Traits

1. **Direct and conversational.** Write like you're explaining to someone sitting next to you. No corporate boilerplate.

2. **Dry humor and self-deprecation.** Light touches, not forced. "He lives in Christchurch but we love him anyway" energy.

3. **Parenthetical asides.** Use parens for commentary, qualifications, and editorial color: "(ie never)", "(still unsolved)".

4. **Honest about mess.** Don't sugarcoat. If something failed, say so plainly. Mark open questions directly.

5. **Concrete over abstract.** Real examples, real names, real systems. "qwen3's thinking mode burns 30+ minutes on large inputs" not "significant performance challenges were encountered."

6. **Structured but not formal.** Headers and bullets for organization, casual prose within them.

7. **Opinionated and specific.** State positions. Attribute them when they belong to others.

### Anti-patterns (DO NOT)

- "In this session, we successfully..." — passive, padded
- "Several challenges were encountered and resolved" — vague, corporate
- "The implementation was completed" — bloodless
- Trailing summaries restating what was just said
- Hedging language ("it appears that", "it seems like")

### Target voice example

> Got Ollama and OpenOats running, transcribed a 96-minute Zoom call. The pipeline works but qwen3's thinking mode is a trap — it burns 30+ minutes on large inputs if you don't append /no_think. Whole-document passes don't scale past ~19K tokens (still unsolved).

## Trigger Mechanisms

### Session-end hook

A Claude Code hook that fires when a session ends. It launches a new Claude Code session that runs the `/journal` skill, passing the session ID of the just-completed session.

Implementation details TBD in planning phase: how to access session transcript data from `~/.claude`, how to identify which session just ended, how to avoid journaling the journal-writing session itself.

### On-demand `/reflect`

User types `/reflect` in any Claude Code session. The skill:
1. Reads all session journals in `Sessions/`
2. Reads existing thread files in `Threads/`
3. Groups sessions by their `threads` frontmatter
4. Creates new thread documents or updates existing ones
5. Optionally accepts a scope argument: `/reflect weekly` to force a weekly digest

### Scheduled weekly digest

A cron job (or Claude Code scheduled agent) that runs weekly. It:
1. Reads all session journals from the past week
2. Reads current thread state
3. Writes `Weekly/YYYY-WNN.md`

## Migration

The six existing journal files in `~/obsidian/Agent Journals/` need to be moved to `Sessions/` and have their frontmatter updated to match the new format (adding `threads`, `tools_installed`, `tags` fields). The `/reflect` skill can then be run to bootstrap the initial thread documents.

## Scope Boundaries

- **Independent from Claude Code memory.** Journals and memory serve different audiences (human review vs. agent operational context). No cross-pollination for now.
- **No Dataview queries shipped.** The frontmatter supports Dataview, but writing specific queries is out of scope.
- **No monthly digests.** Weekly is the only scheduled temporal reflection. Monthly can be added later if weekly proves valuable.
